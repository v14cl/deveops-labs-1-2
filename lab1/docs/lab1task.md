# Лабораторна робота №1

## Розгортання Web-сервісу з автоматизацією

## Мета роботи

Метою роботи є практичне закріплення знань з таких тем:

* основи Linux-адміністрування
* мережі та мережеві сервіси
* використання Git
* скриптування (Bash або Python)
* керування сервісами через systemd

Необхідно розробити та розгорнути простий web-застосунок на віртуальній машині Linux, автоматизувати його встановлення та налаштування, а також реалізувати обмежене адміністрування сервісів.

---

# Індивідуальне завдання

## Загальна архітектура системи

Усі компоненти повинні бути розгорнуті на одній віртуальній машині Linux.

Архітектура системи:

client → nginx (reverse proxy) → web application → PostgreSQL

---

## Мережеві параметри

| Компонент       | Адреса    | Порт                        |
| --------------- | --------- | --------------------------- |
| nginx           | 0.0.0.0   | 80                          |
| web application | 127.0.0.1 | 8000                        |
| PostgreSQL      | 127.0.0.1 | стандартний PostgreSQL-порт |

---

## Вимоги до мережі

* доступ до PostgreSQL має бути дозволений лише локально з віртуальної машини
* клієнти повинні взаємодіяти із застосунком лише через nginx reverse proxy
* web-застосунок не повинен бути доступний напряму із зовнішньої мережі

---

# Тематика web-застосунку

## Notes Service

Необхідно реалізувати сервіс для збереження текстових нотаток.

---

## Модель нотатки

Об’єкт нотатки повинен містити поля:

* id
* title
* content
* created_at

---

## API застосунку

### Отримання списку нотаток

GET /notes

Повертає список усіх нотаток:

* id
* title

---

### Створення нової нотатки

POST /notes

Параметри:

* title
* content

---

### Отримання повної інформації про нотатку

GET /notes/<id>

Повертає:

* id
* title
* created_at
* content

---

# Назва застосунку

Назва web-застосунку незалежно від варіанту:

mywebapp

---

# Конфігурація застосунку

Застосунок повинен конфігуруватися через конфігураційний файл:

/etc/mywebapp/config.json

Файл конфігурації має містити:

* параметри підключення до PostgreSQL
* порт застосунку
* інші необхідні налаштування

Формат конфігурації — JSON.

---

# Порт застосунку

Web-застосунок повинен працювати локально на порту:

8000

---

# База даних

Для збереження даних необхідно використовувати PostgreSQL.

База даних повинна бути доступна лише локально через:

127.0.0.1

---

# Реалізація web-застосунку

Для реалізації дозволяється використовувати будь-яку мову програмування та framework.

---

## Обов’язкові ендпоінти стану сервісу

### Перевірка працездатності

GET /health/alive

Повинен:

* завжди повертати HTTP 200
* повертати текст:

OK

---

### Перевірка готовності сервісу

GET /health/ready

Повинен:

* повертати HTTP 200 та OK, якщо застосунок успішно підключився до PostgreSQL
* повертати HTTP 500, якщо сервіс не готовий до роботи

---

# Формати відповіді API

API повинно підтримувати:

* text/html
* application/json

---

## Вимоги до HTML-відповідей

Якщо клієнт надсилає:

Accept: text/html

необхідно:

* повертати прості HTML-сторінки
* не використовувати JavaScript
* не використовувати CSS
* списки даних відображати таблицями

---

## Вимоги до JSON-відповідей

Якщо клієнт надсилає:

Accept: application/json

необхідно повертати JSON.

---

# Кореневий ендпоінт

Необхідно реалізувати:

GET /

Ендпоінт повинен:

* працювати лише з text/html
* повертати HTML-сторінку зі списком усіх бізнес-ендпоінтів застосунку

---

# Міграція бази даних

Необхідно реалізувати скрипт міграції PostgreSQL.

Скрипт повинен:

* створювати всі таблиці
* створювати індекси
* коректно працювати як із порожньою БД, так і з попередньою версією структури

---

# Розгортання сервісу

Необхідно розгорнути:

* web-застосунок
* PostgreSQL
* nginx reverse proxy

на Linux-віртуальній машині.

---

# Systemd для web-застосунку

Необхідно створити systemd unit:

/etc/systemd/system/mywebapp.service

---

## Вимоги до systemd-сервісу

Сервіс повинен:

* запускатися автоматично через systemd
* працювати від окремого користувача mywebapp
* перед запуском виконувати міграцію БД

Після реалізації звичайного service unit необхідно переробити запуск на:

systemd socket activation

---

# Reverse Proxy

Необхідно налаштувати nginx як reverse proxy.

---

## Вимоги до nginx

nginx повинен:

* слухати порт 80
* проксувати запити до web-застосунку на 127.0.0.1:8000
* вести access logs
* назовні віддавати лише:

    * кореневий ендпоінт
    * бізнес-ендпоінти застосунку

---

# Користувачі системи

Необхідно створити користувачів:

| Користувач | Призначення         | Права                |
| ---------- | ------------------- | -------------------- |
| student    | робота з проєктом   | sudo/root            |
| teacher    | перевірка роботи    | sudo/root            |
| app        | запуск застосунку   | мінімально необхідні |
| operator   | керування сервісами | обмежений sudo       |

---

## Паролі

Для користувачів:

* student
* teacher
* operator

встановити пароль:

12345678

Після першого входу пароль повинен бути змінений.

---

# Обмеження для користувача operator

Користувач operator повинен мати право лише на:

* запуск web-застосунку
* зупинку web-застосунку
* перезапуск web-застосунку
* перегляд статусу web-застосунку
* reload nginx

---

# Автоматизація розгортання

Необхідно реалізувати автоматичне розгортання системи за допомогою Bash або Python-скриптів.

Повинна існувати одна точка входу для запуску автоматизації.

---

## Скрипт автоматизації повинен:

* встановлювати необхідні пакети
* створювати користувачів
* створювати PostgreSQL базу даних
* копіювати або генерувати конфігураційні файли
* встановлювати systemd unit
* запускати web-застосунок
* налаштовувати nginx
* створювати файл:

/home/student/gradebook

який містить число:

3

---

## Додаткова вимога

Після завершення автоматизації необхідно:

* заблокувати дефолтного користувача системи
* залишити можливість входу лише для:

    * student
    * teacher
    * operator

---
